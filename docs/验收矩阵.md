# 发布验收矩阵

本文件是 CI 与中文操作手册之间的导航页。稳定 acceptance ID 的唯一权威顺序是
`scripts/verify_evidence_contract.py:ACCEPTANCE_NAMES`，当前固定为 **140 项**；
报告必须保持相同顺序，不能用一个总分替代独立 item。

## CI 分层

触及产品、测试、依赖、验收脚本、schema 或 workflow 的 push/PR 才运行
`.github/workflows/fast-verify.yml`；纯 `docs/**/*.md`、截图和审阅记录提交不会触发
它。目标是每个 job 十分钟内完成 Windows/Linux 的高价值真实功能门禁。Stage 1 执行源码态 `182/182 self-check`、
`140/140 GUI acceptance`（1280x720）、聚焦回归测试，并构建一个轻量 onefile；
Stage 2 在不 checkout、不安装项目 Python 依赖的 fresh runner 上下载该 onefile，
只执行 `182/182 self-check`，证明冻结产物的运行时依赖闭包。它不重复完整 GUI
acceptance 矩阵，也不构建 onedir 或三平台发布包。

完整三平台产物证据由 `.github/workflows/build.yml` 提供，触发方式为 GitHub Actions
的 `workflow_dispatch` 或 `v*` tag。它才运行 Linux/Windows/macOS 的 source、
onedir、onefile、fresh runner 矩阵，并上传 12 个产品/evidence artifact。文档或小型
GUI 修复不应反复触发这个重型发布流程。

| 门禁 | Linux | Windows | macOS | 触发 | 证明内容 |
| --- | ---: | ---: | ---: | --- | --- |
| Fast Verify / Stage 1 build | 必须 | 必须 | 不参与默认 push | push/PR | source self-check、140 GUI acceptance、聚焦回归、轻量 onefile |
| Fast Verify / Stage 2 fresh | 必须 | 必须 | 不参与默认 push | push/PR | fresh onefile self-check（182 项；无项目 Python 依赖） |
| Full Release / Stage 1 | 必须 | 必须 | 必须 | 手动/tag | 原生 source + onedir + onefile、模板编译运行 |
| Full Release / Stage 2 | 必须 | 必须 | 必须 | 手动/tag | fresh runner 黑盒 self-check + GUI acceptance |

Fast Verify 的 fresh 阶段只证明 priority Windows/Linux onefile 的 self-check 运行闭包，
不是完整 GUI fresh acceptance 或三平台发布包自包含证明；发布包发布前仍必须运行 Full Release。反过来，
Full Release 的长耗时不应阻塞普通文档迭代。

### 产品 lineage 复用规则

当提交只改变手册、截图、审阅记录或不影响产品行为的证据脚本，并且
`git diff --quiet <已验收产品 SHA>..HEAD -- app main.py main.spec requirements*.txt Makefile`
通过时，历史 Full Release 的产品/evidence/visual attestation 可以作为当前 main 的
lineage-equivalent 证据。此时必须保留历史产品 SHA、run ID 和 verifier 结果，状态写成
`PASS-LINEAGE-ATTESTED`，不能写成 current-SHA fresh。任何产品树、依赖或打包边界变化都
立即取消该复用资格，重新运行完整三平台 Full Release。

## Self-check

- `114` 个 pyfcstm module-closure 项与 `68` 个 behavior 项分开计数，总计 `182`。
- Z3 独立执行 integer SAT/model、UNSAT、exact real、bit-vector、Optimize 五项，
  每项真实 `check()` 并断言模型或最优值。
- loader、inspect、公式、SimulationRuntime 多 cycle/rollback、四个动态场景、
  mutation/recover、PlantUML Java、Office 文件、Qt 原生控件和五套模板均执行逻辑，
  不能只 import 或读取版本号。

## 视觉与阻塞判定

仓库内 `docs/images/workflows/` 是 Linux xcb source-reference，只负责图解操作；
不能冒充 fresh release evidence。发布前从 Full Release 下载同一 run 的代表图，按
`docs/visual-review.schema.json` 形成 54 项（3 平台 × 2 布局 × 9 图）attestation。

阻塞只由功能事实决定：文字/数据裁切、误触、焦点不可达、缺 accessible name、
业务或文件/JSON/SHA 错误是 blocking。原生 focus ring、边界接触和轻微留白，只要
七项功能 verdict 全真且人工确认不妨碍使用，就记录为 `non_blocking_findings`。

## 本地复现入口

```bash
# 快速功能门禁（Linux 示例）
QT_QPA_PLATFORM=xcb xvfb-run -a .venv/bin/python main.py --self-check \
  --json-report /tmp/fcstm-self-check.json
QT_QPA_PLATFORM=xcb xvfb-run -a .venv/bin/python main.py --acceptance-check \
  --viewport 1280x720 --json-report /tmp/fcstm-acceptance.json \
  --artifact-dir /tmp/fcstm-acceptance-artifacts

# 完整源码回归（慢平台给足外层预算；单项 GUI 等待仍有界）
.venv/bin/python -m pytest test -q --timeout=600
```

发布包用户不需要 Python、Graphviz 或编译器；图形 PNG/SVG/PDF 只需要 Java 11 或
兼容 JRE。当前矩阵不构成 Windows 7 实机兼容性声明。
