# fcstm-gui 完整操作与验收手册

本文档用于让未阅读源码的操作者从零完成 fcstm-gui 的全部 GUI 验收，并留下可复核证据。步骤中的菜单名、按钮名、状态和产物都是验收合同；控件缺失、文案不符、操作必须绕过 GUI、结果无法复核时，均应记录失败，不能自行跳步。

## 0. 证据边界

- 本文图片由源码态应用在 Linux x86_64、xcb、1280x720 logical px、scale 1 下真实重放生成。
- 图片使用内置 `Noto Sans CJK SC` 10 pt，详情见 [manifest.json](images/workflows/manifest.json)。
- 140 个稳定 acceptance ID 的逐项实操截图、详情、revision/fingerprint 和 SHA 见
  [140 项 GUI Acceptance 截图索引](images/acceptance-140/README.md)，其机器可读合同见
  [acceptance gallery manifest](images/acceptance-140/manifest.json)。
- `images/workflows/` 是按用户旅程整理的分阶段参考图；`images/acceptance-140/` 是
  每个稳定 ID 的独立结果图。两者都来自源码态 Linux xcb，不能冒充其他平台 fresh。
- 这些图片是操作手册参考图，不得替代 fresh onedir/onefile 最终发布证据。
- 最终 READY 仍要求 Linux、Windows、macOS fresh onedir/onefile 的 acceptance JSON、截图、manifest、artifact digest 和 Issue visual attestation 全部通过。
- 图片中的 `<TEMP>`、`<WORKSPACE>` 为正常脱敏；不要为了截图暴露真实用户路径。
- 视觉问题是否阻塞只由功能决定：只要不妨碍使用，且文字/数据可读、控件中心 hit-test 正确、真实点击流程通过、键盘焦点可达、accessible name/tooltip 完整、业务事实和产物正确，原生 style 外框或 focus ring 接触就记录为 non-blocking，不得阻塞交付。任一功能事实失败或未知遮挡仍为 blocking。
- 同步操作没有可观察 running 状态时，步骤图矩阵写 `N/A（同步完成，无可观察 running 态）` 和原因；不得复制结果图冒充 running。

### 0.1 当前 CI 使用方式

触及 `app/`、入口/spec、依赖、`scripts/`、`test/`、schema 或 workflow 的 push/PR
才触发 `.github/workflows/fast-verify.yml`；纯手册、截图和审阅记录提交不会触发
现有 verify/test。Windows/Linux 每个 job 都有十分钟预算，分成两个阶段：Stage 1 在源码态真实执行 `182/182 self-check`、
`140/140 GUI acceptance`（1280x720）和聚焦回归测试，并构建一个轻量 onefile；
Stage 2 使用 fresh runner 下载该 onefile，不 checkout、不安装项目 Python 依赖，
只执行 onefile 的 `182/182 self-check`。它不重复 GUI acceptance，因此文档和小型
GUI 修订不会反复等待完整三平台发布矩阵。

需要发布包或重新取得三平台 fresh 证据时，在 Actions 手动运行
`Full Release Build & Verify`，或推送 `v*` tag。该流程才执行 source/onedir/onefile
和三平台 fresh runner 全矩阵；它的 artifact 是完整发布证据。Fast Verify fresh
artifact 只覆盖 Windows/Linux onefile self-check，不要把它当作完整发布证据。
若只修改手册、截图、审阅记录或 workflow/证据脚本，且
`git diff --quiet b6bb7e2..HEAD -- app main.py main.spec requirements*.txt Makefile`
通过，则沿用历史 Full Release baseline，不重复消耗三平台发布矩阵；一旦核心产品树
变化，必须重新取得对应 fresh artifact。

## 1. 前置条件

### 1.1 普通用户

1. 解压同一平台的完整 onedir ZIP，或取得 onefile 成品。
2. 准备 Java 11 或兼容 JRE，并执行 `java -version`。
3. 不安装项目 Python、requirements、Graphviz 或编译器来帮助 fresh 成品通过。
4. runner 预装 Python 仅可在成品退出后运行仓库随 evidence 携带的纯标准库
   manifest/SHA 校验器；不得 import 项目、安装包或参与成品功能执行。
5. 准备一个可写的空目录，例如 `acceptance-output`。
6. 启动 `fcstm-gui`，确认首屏有“导入状态机”和“新建状态机”。

### 1.2 源码态文档截图维护者

在仓库根目录执行：

```bash
QT_QPA_PLATFORM=xcb QT_SCALE_FACTOR=1 \
  xvfb-run -a .venv/bin/python scripts/capture_workflow_docs.py \
  --viewport 1280x720 --scale 1
```

脚本会真实执行打开、诊断、Smetana 图形、普通仿真、动态验证、五模板生成、统一导出和任务历史，并重建 `docs/images/workflows/`。运行后必须逐张打开图片，复算 manifest SHA，再更新第 8 节审阅记录。

若已有一次完整 acceptance report，可把 140 项截图固化为逐项图廊：

```bash
venv/bin/python -m scripts.build_acceptance_gallery \
  --report /path/to/acceptance-output/report.json \
  --artifacts /path/to/acceptance-output/artifacts \
  --source-commit "$(git rev-parse HEAD)" \
  --output docs/images/acceptance-140
```

命令拒绝非 `140/140 passed`、顺序不等于冻结 `ACCEPTANCE_NAMES` 或缺失 item PNG 的报告。

### 1.3 本手册固定模型

核心流程使用以下 `acceptance.fcstm`：

```fcstm
def int count = 0;
state Root {
    state Idle;
    state Running;
    [*] -> Idle;
    Idle -> Running :: Start effect { count = count + 1; }
    Running -> [*] :: Stop;
}
```

验收事实：初始状态为 `Root.Idle`；输入 `Start` 后进入 `Root.Running` 且 `count=1`；输入 `Stop` 后结束。

### 1.4 可复现 fixture 包

在 `acceptance-output` 旁新建 `acceptance-work`。所有文本文件均以 UTF-8 保存；除明确要求 CRLF 的文件外使用平台默认换行。源码仓库操作者可直接使用下列已入库资源，成品操作者则按本节文本创建同名文件。不得临时改变 expected 来迎合 actual。

| 文件/目录 | 创建内容或仓库来源 | 用途 |
| --- | --- | --- |
| `acceptance.fcstm` | 第 1.3 节完整文本 | 主流程 |
| `other.fcstm` | `state Other;` | dirty/open 替换 |
| `warning.fcstm` | `state Root { state A; [*] -> A; }` | inspect warning `W_DEADLOCK_LEAF` |
| `assembly.fcstm` | `state Root { state A; [*] -> Missing; }` | assembly/model 缺失目标 |
| `recent-01.fcstm` ... `recent-12.fcstm` | 分别复制 `state Recent01;` ... `state Recent12;` | 最近文件上限与排序 |
| `leaf.fcstm` | `state Leaf;` | imported 来源叶文件 |
| `child.fcstm` | `state Child { import "./leaf.fcstm" as Nested; [*] -> Nested; }` | imported 中间文件 |
| `root-import.fcstm` | `state Root { import "./child.fcstm" as Imported; [*] -> Imported; }` | imported 根文件 |
| `rename-crlf.fcstm` | 下方代码块，编辑器明确选择 CRLF | Unicode/CRLF 重命名 |
| `dynamic-model.fcstm`、`dynamic-pass.json`、`dynamic-mismatch.json` | 下方动态验证块 | user/mutation/recover |
| `custom-template/config.yaml` | 仅一行 `{}` | 自定义模板配置 |
| `custom-template/hello.txt.j2` | `root={{ model.root_state.name }}` | 自定义模板非空产物 |

`rename-crlf.fcstm` 内容：

```fcstm
// 😀 中文：state Parent Leaf 仅为注释，必须逐字保留
state Root {
    // state Parent 也仅为注释
    state Parent {
        state Leaf;
        state Other;
        [*] -> Leaf;
        Leaf -> Other;
    }
    [*] -> Parent;
}
```

动态验证文件以仓库资源
`app/resources/self_check/dynamic_validation/design_validation_failure_multilevel_transition.{fcstm,json}`
为权威来源。成品盲测时复制该 `.fcstm` 为 `dynamic-model.fcstm`，复制 JSON 为
`dynamic-pass.json`，仅把 JSON 的 `model_file` 改为 `dynamic-model.fcstm`。再复制为
`dynamic-mismatch.json`，把 `case_id` 改成 `manual_mutation`，并且只把最后一步
`expected.state` 从 `Root.A` 改为 `Root.Mutated`。两个 JSON、模型必须放在同一目录；
除这两个字段外逐字保持。这样 pass、mutation、restore 使用同一生产 loader 和 runtime。

`dynamic-model.fcstm` 完整内容：

```fcstm
def int counter = 0;
def int flag = 0;
state Root {
    state A { during { counter = counter + 1; } }
    state B {
        enter { flag = 1; }
        state B1 { during { counter = counter + 10; } }
        [*] -> B1 : if [flag == 2];
    }
    [*] -> A;
    A -> B : if [counter >= 3] effect { flag = 1; };
}
```

`dynamic-pass.json` 完整内容（文件名字段已经改好）：

```json
{
  "schema": "fcstm-gui.dynamic-validation-scenario",
  "version": 1,
  "case_id": "design_validation_failure_multilevel_transition",
  "upstream_case_id": "design_validation_failure_multilevel_transition",
  "upstream_commit": "f142a656df43e0b80ed6b7ac63b0696345325646",
  "model_file": "dynamic-model.fcstm",
  "initial": {"state": null, "variables": {}},
  "steps": [
    {"events": [], "commands": [], "expected": {"state": "Root.A", "variables": {"counter": 1, "flag": 0}, "ended": false}},
    {"events": [], "commands": [], "expected": {"state": "Root.A", "variables": {"counter": 2, "flag": 0}, "ended": false}},
    {"events": [], "commands": [], "expected": {"state": "Root.A", "variables": {"counter": 3, "flag": 0}, "ended": false}},
    {"events": [], "commands": [], "expected": {"state": "Root.A", "variables": {"counter": 4, "flag": 0}, "ended": false}}
  ]
}
```

每轮开始前记录这些文件的 SHA-256。若操作者无法取得仓库资源，则必须从本手册附带的
完整代码块创建；仓库资源用于复算权威 SHA。任何代码块、资源或 release fixture 缺失都属于文档/发布失败，不能跳过对应 family。

## 2. 核心流程

### 2.1 打开文档

**前置条件**：应用刚启动，尚未打开文档。

1. 确认首屏仅显示真实空状态，不显示虚假的模型、诊断或任务成功记录。
2. 按 `Ctrl+O`；macOS 按 `Cmd+O`。
3. 在文件选择器中输入 `acceptance.fcstm` 的路径并确认。
4. 等待文档状态区显示文件名、`revision 0`、已保存和校验状态。
5. 确认模型树出现 `Root / Idle / Running`，属性检查器显示根状态来源。

![打开前](images/workflows/01-open-document/00-before.png)

![打开结果](images/workflows/01-open-document/03-result.png)

**预期状态**：当前 session 指向所选文件；源码文本逐字等于文件；有效 snapshot 可用；最近文件菜单增加该文件。

**失败恢复**：文件不存在、编码无法确定或依赖加载失败时，旧 session、dirty、选择、undo 栈和当前工作区必须保持。不要通过重新启动掩盖失败。

失败保持使用确定步骤：先成功打开 `other.fcstm` 让它进入最近文件，返回 `acceptance.fcstm` 并制造一个未保存空格；在操作系统中删除 `other.fcstm`，再从“文件 -> 最近文件”选择它。记录失败前后的 session URI、source revision、dirty、selection、undo 可用性和 current workspace，必须逐项相同；该项为 `document.failed-load-preserves-session`。重新创建 `other.fcstm` 后从最近文件重开为 `document.recent-reopen`。加载运行行出现时通过任务中心动作列请求取消并核对同样事实，为 `document.cancel-load`；如果任务过快未出现 running 行，该次不算取消通过。

**自动化 item**：`document.open`、`keyboard.workspace`、`tasks.registry.load`。

### 2.2 结构化诊断、定位与恢复

**前置条件**：已打开固定模型。

1. 切换“源码”页，把内容替换为 `state Broken { state ; }`。
2. 等待防抖校验结束，确认状态为“语法无效”，图形、仿真、生成和导出被禁用。
3. 切换“检查”页。
4. 在诊断表核对来源 `syntax`、消息、真实位置和“定位”按钮。
5. 选中首行并按“定位”；焦点必须回到源码错误范围，且范围非空。
6. 恢复第 1.3 节源码，等待完整重验。
7. 返回检查页，确认旧诊断不再作为当前 revision 结论。

三来源必须分别执行，不能用 syntax 一项代替：

| 稳定 ID | 确定输入 | 必须观察 |
| --- | --- | --- |
| `diagnostics.syntax` | 源码替换为 `state Broken { state ; }` | source=`syntax`，非空 span，定位选择错误 token |
| `diagnostics.assembly` | GUI 打开 `assembly.fcstm` | source=`model/assembly`，消息含 `Missing`，旧有效 snapshot 不冒充当前 |
| `diagnostics.inspect` | GUI 打开 `warning.fcstm` 后按 F5 | warning、code=`W_DEADLOCK_LEAF`、span 和 provenance 均可见 |
| `diagnostics.conflict-warning` | 在 `warning.fcstm` 增加第二个同名 `state A;`，恢复后再 F5 | 冲突阻止提交；恢复后 warning 可筛选和搜索 |
| `diagnostics.suggested-fix` | 源码固定为 `state Root { state A; [*] -> A; }`，按 F5，选择 code=`W_DEADLOCK_LEAF` | “建议修复”必须可见；确认预览后源码精确增加 `A -> [*];`，revision 增加并重新验证通过 |

每项依次操作严重度筛选、文本搜索、详情、定位；保存 screenshot、当前 revision、URI、code 和 suggested-fix 结构。恢复固定模型后再进入下一项。

![有效源码](images/workflows/02-diagnostics-navigation/00-before.png)

![结构化诊断](images/workflows/02-diagnostics-navigation/01-action.png)

![定位源码](images/workflows/02-diagnostics-navigation/03-result.png)

![恢复有效](images/workflows/02-diagnostics-navigation/04-failure-recovery.png)

**预期状态**：诊断绑定当前 session/revision/fingerprint；恢复后消费者重新可用；旧有效模型不会冒充无效源码的当前模型。

**产物/schema**：inspect JSON 使用版本化对象；每项保留 severity、source kind、code、message、span、URI 和 provenance。

**自动化 item**：`diagnostics.syntax`、`diagnostics.assembly`、`diagnostics.inspect`、`diagnostics.filter-search`、`diagnostics.conflict-warning`、`diagnostics.locate`、`diagnostics.suggested-fix`、`formula.stale`。

### 2.3 真实状态图

**前置条件**：当前 revision 有有效 snapshot，Java 可用。

1. 按 `Ctrl+3` 或 `Cmd+3` 打开“图形”页。
2. 按“刷新”。任务运行时状态应立即变为渲染中，但任务 dock 不应因普通成功任务强制展开。
3. 等待状态变为“就绪”。
4. 在画布中确认 `Root`、`Idle`、`Running`、迁移标签 `Idle.Start` / `Running.Stop` 与变量表真实可见；“适应”缩小时若笔画发虚，按“100%”并滚动查看完整标签。
5. 依次操作“适应”“100%”“重置”，滚轮缩放并拖动画布。
6. 在左侧模型树单击 `Idle`，确认图形状态行显示 `选择 Root.Idle`；切回图形页后选择仍保留。这是 `graph.selection`。
7. 确认画面中不存在 `Cannot find Graphviz`、`dot executable` 或 renderer error。

![刷新前](images/workflows/03-real-state-graph/00-before.png)

![渲染中](images/workflows/03-real-state-graph/02-running.png)

![真实 Smetana 图](images/workflows/03-real-state-graph/03-result.png)

**预期状态**：engine 为 `smetana`；renderer exit code 为 0；SVG 语义与同一 PlantUML source SHA 绑定；错误图不得覆盖上一张有效图。

**图形导出**：在图形页“导出”按钮左侧的类型下拉框依次选择下表类型，再按“导出”；目标分别为
`graph.puml`、`graph.png`、`graph.svg`、`graph.pdf`。每次从当前 ready 图重新执行，不复用统一导出的文件。

| ID | GUI 选择 | 文件 oracle |
| --- | --- | --- |
| `graph.export.plantuml` | 导出 -> PlantUML | UTF-8；含 `@startuml`、Smetana pragma、Root/Idle/Running；SHA 与当前图 source SHA 一致 |
| `graph.export.png` | 导出 -> PNG | 前 8 bytes 为 PNG magic；宽高大于 0；无 renderer error 文本 |
| `graph.export.svg` | 导出 -> SVG | 可解析 XML；根元素为 `svg`；文本含 Root/Idle/Running |
| `graph.export.pdf` | 导出 -> PDF | 前 5 bytes 为 `%PDF-`；文件非空；任务产物绑定当前 revision/source SHA |

为每项保存目标 SHA-256、size、magic/解析结果和独立截图。图形页没有覆盖开关；先放置内容为 `OLD` 的同名文件后再次导出，必须失败且旧 SHA 不变。需要替换时由用户另选新文件名，不得静默覆盖。

**失败恢复**：渲染失败时显示结构化原因和“重试当前版本”，保留上一张有效图并标明旧 revision。

**自动化 item**：`graph.refresh`、`graph.zoom`、`graph.drag`、`graph.fit`、`graph.reset`、`graph.selection`、`graph.export.plantuml`、`graph.export.png`、`graph.export.svg`、`graph.export.pdf`、`cancel.graph`、`stale.graph`。

### 2.4 普通仿真

**前置条件**：当前 revision 有有效 snapshot。

1. 按 `Ctrl+5` 或 `Cmd+5` 打开“普通仿真”。
2. 初始状态留空，初始变量保持 `{}`，按“初始化”。
3. 核对状态为“就绪”、当前状态为 `Root.Idle`、cycle 为 1、`count=0`。
4. 在“输入事件”填写 `Start`，按“单步”。
5. 核对 `Root.Running`、cycle 2、`count=1`，以及已消费/未消费事件和结果列。
6. 把“连续 cycle 上限”设为 `10000`，按“连续运行”；“暂停”启用后立即按下。
7. 等待 cycle 边界显示“已暂停”，再按“继续”；确认 runtime、cycle 和 transcript 连续。
8. 继续时把上限改为 `2`；结果 cycle 必须恰好再增加 2，runtime identity 和旧 transcript 保持。
9. 再次把上限设为 `10000` 并连续运行，“停止”启用后立即按下；状态必须为“已取消，已保留完成的 cycle”，不得显示 success/failed。
10. 记录取消后的 transcript 行数，按“重置”；确认同一文档重新回到 `Root.Idle`、cycle 1、`count=0`，旧取消记录仍可在任务历史查看。

![初始化](images/workflows/04-ordinary-simulation/01-action.png)

![单步结果](images/workflows/04-ordinary-simulation/03-result.png)

暂停图必须同时显示“已暂停”和“继续运行”，且 transcript 保留暂停前已经完成的 cycle。

![同一 runtime 继续后的结果](images/workflows/04-ordinary-simulation/05-simulation-continued.png)

**预期状态**：暂停 segment 以 success/outcome=`paused` 结束，继续创建新 segment 但复用同一 runtime；停止为 cancelled；修改源码后下一个 cycle 边界变为 stale 并保留已完成 transcript。

**自动化 item**：`simulation.initialize`、`simulation.step`、`simulation.run`、`simulation.pause`、`simulation.continue`、`simulation.reset`、`simulation.stop`、`cancel.simulation`、`stale.simulation`、`keyboard.simulation`。

### 2.5 动态验证

**前置条件**：当前 revision 有有效 snapshot。动态验证与普通仿真必须是独立 runtime。

1. 按 `Ctrl+6` 或 `Cmd+6` 打开“动态验证”。
2. 先读页面固定声明：“动态验证是场景 expected/actual 对比，不是形式化验证、模型检测或证明”。
3. 按“全部验收”，等待四个内置用例完成。
4. 确认总状态“通过”（结构化 JSON status=`passed`），表中逐 step 显示 expected、actual、diff、rollback 和结果。

## 3. 模型编辑与公式验收

以下每一行都必须通过真实表单完成一次 add/edit/delete，并保存后关闭、重新打开文件核对事实。不要直接改 Python model。

| 对象 | 新增步骤 | 编辑步骤 | 删除步骤 | 必须核对 |

| 状态 | 模型页选择父状态 -> 模型/新建状态 -> 输入唯一名称 -> 提交 | 选择状态 -> 编辑名称 -> 预览 TextEdit -> 提交 | 选择无冲突状态 -> 删除 -> 确认 | 注释、空白、子状态和迁移未被整段重写 |

| 迁移 | 选择状态 -> 新建迁移 | 编辑 source/target/event/guard/effect | 删除并确认 | SourceRef 精确、整模重验通过 |
| guard | 迁移对话框填写 logical 表达式 | 合法与非法各一次 | 清空可选 guard | 字段位置、reason、防抖和提交门禁 |

| `cancel.graph` | 刷新后立即停止 | 上一张有效图保留；不得发布诊断/半图 |
| `cancel.generation` | 使用隔离 custom template 目录（确定性生成 300 个小模板文件）后启动生成，running 行出现即停止 | staging 清理；旧目标 inventory/SHA 不变 |
| `cancel.export` | 使用固定图形 fixture 和确定性 PNG/PDF 目标，running 行出现即停止 | 临时文件清理；旧目标 bytes/SHA 不变 |

所有结果记录取消边界、完成数量、临时目录 inventory、旧目标 SHA 和 TaskStatus=`cancelled`。任务过快导致三次均无法命中取消窗口，应记录产品可测性失败，不能把普通完成算作通过。

人工点击受机器调度影响时，直接执行发布包内置的确定性取消/stale 验收；它会生成足够大的隔离 fixture，并在控件启用后立即点击，不需要修改源码或安装 Python：

```bash
./fcstm-gui --acceptance-check --viewport 1280x720 \
  --json-report acceptance-output/report.json \
  --artifact-dir acceptance-output/artifacts
```

onedir 用户先 `cd fcstm-gui` 再执行上式；onefile 用户在二进制所在目录直接执行。Windows PowerShell 对 onedir 同样先进入 `fcstm-gui` 目录，再把命令开头改为 `.\fcstm-gui.exe`。命令必须最终显示 `140 passed / 0 failed`；在 `report.json` 中逐项确认六个 `cancel.*` 为 passed、`evidence.status=cancelled`，且各项 `artifact_inventory` 非空。普通完成不得替代取消结果。

### 4.4 五类 stale

对 `stale.graph`、`stale.simulation`、`stale.dynamic`：启动任务，确认 busy/停止已启用，立即切到源码末尾输入一个空格并等待 revision 增加；仿真在 cycle 边界、动态验证在 step 边界、图形在发布前检查 stale。

`stale.generation`、`stale.export` 当前由发布包内置 acceptance harness 以 **modeless
确定路径** 验证：harness 打开生成/导出对话框后，在任务仍运行时对隔离 fixture
追加源码变更，重新计算 dependency fingerprint，再断言旧结果被拒绝。普通菜单中的
生成/导出对话框是 modal，不能把“同时点击被 modal 阻塞的源码页”当作普通用户手工
路径；若要手工复现，应先用外部编辑器修改 harness 指定的 `leaf.fcstm`，然后重新
从 GUI 提交任务，而不是声称当前产品有文件 watcher 自动触发 stale。

统一 oracle：TaskStatus=`stale`，结果保存旧 revision/fingerprint 供查看，但当前图、runtime/report、生成目录或导出目标不得被旧结果覆盖。随后 undo 源码空格，或把 `leaf.fcstm` 恢复成原 bytes，再重跑并确认成功绑定新 revision/fingerprint。每项记录 before/after revision、fingerprint、目标 SHA 和 task id。

![stale 图形显示已失效且不发布旧图](images/workflows/11-cross-cutting/03-stale-graph.png)

同一份内置 acceptance 报告还必须包含五个 `stale.*` passed 项；逐项核对旧 revision/fingerprint、`status=stale` 和未发布旧产物的 evidence。这样人工调度速度不会成为 stale 证明的前提。

自动化 item：`document.recent-reopen`、`dirty.save`、`dirty.discard`、`dirty.cancel`，以及第 6 节展开的六个 cancel 和五个 stale ID。
