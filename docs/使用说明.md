## 0. 桌面工作台

### 0.1 打开与编辑

通过“文件 -> 导入状态机”或 `Ctrl+O` 打开 `.fcstm`。源码是唯一权威内容；
模型树、属性、事件和表单都是当前 revision 的投影。诊断表提供等级、错误码、
来源和位置，“定位”会回到对应源码范围。无效源码不会覆盖最近一次有效快照，
但状态图、仿真、生成和导出只允许使用当前 revision 的有效快照。

### 0.2 状态图、仿真与动态验证

- “图形”页可刷新状态图、适应窗口、恢复 100%，并导出 PlantUML、PNG、SVG、PDF。
- “普通仿真”页先初始化，再输入事件并执行 cycle；异常会显示 cause，并保留 rollback 后状态。
- “动态验证”页可运行用户 JSON 场景、单个内置用例或全部四个内置用例，并导出带 SHA-256、模型 fingerprint 和 revision 的版本化报告。

动态验证是确定性运行场景验收，不是形式化验证。

### 0.3 代码生成与统一导出

“生成 -> 代码生成”支持 `python`、`c`、`c_poll`、`cpp`、`cpp_poll` 五个
内置模板以及自定义模板目录。目标目录采用临时目录生成后原子发布；revision
变化、取消或生成失败不会留下截断产物。

“导出 -> 统一导出”集中提供 DSL、inspect JSON/文本、PlantUML、PNG、SVG、
PDF、XLSX 和 DOCX。默认不覆盖已有目标。

### 0.4 任务结果与隐私

显式任务会显示状态、revision、摘要、详情与产物。复制、持久化和日志导出默认
隐藏工作区、主目录和临时目录；只有主动开启“完整路径”才显示本机绝对路径。

### 0.5 发布包自检

```bash
fcstm-gui --self-check --json-report self-check.json
fcstm-gui --acceptance-check \
  --viewport 1280x720 \
  --json-report acceptance/report.json \
  --artifact-dir acceptance/artifacts
```

两条命令均以退出码作为门禁，JSON schema version 当前为 `1`。CI 还会在
1280×720、1920×1080 与 100%、150%、200% 缩放比例的交叉组合中运行 GUI
验收。

## 1. 安装与使用

### 1.1 安装

本工具以 ZIP 包形式提供，解压后即可通过内置的可执行文件直接运行，无需额外安装。

### 1.2 基本使用

对于已经建模好的dsl程序，可以生成代码或导出PluntUML格式。

**代码生成：**

```
pyfcstm generate -i <输入DSL文件> -t <模板目录> -o <输出目录> [--clear]
```

**参数说明：**

- -i 或 --input-code：输入的 DSL 文件路径（如 example.fcstm）

- -t 或 --template-dir：代码生成模板目录

- -o 或 --output-dir：生成代码的输出目录

- --clear：可选，生成前清空输出目录

**示例：**

```
pyfcstm generate -i example.fcstm -t ./template_1 -o ./output
```

**导出PlantUML格式：**

支持导出PlantUML格式，自动生成状态机图

```
pyfcstm plantuml -i <输入DSL文件> -o <输出PlantUML文件>
```

**参数说明：**

- -i 或 --input-code：输入的 DSL 文件路径

- -o 或 --output-file：导出的 PlantUML 文件路径

**示例：**

```
pyfcstm plantuml -i example.fcstm -o example.puml
```

## 2. 文件结构与关键字

### 2.1 文件结构

一个完整的 DSL 程序由若干可选的变量定义和一个根状态定义组成：

```fcstm
program ::= def_assignment* state_definition EOF
```

这一顶层结构保证了每个状态机都有且仅有一个根状态，该根状态可以包含嵌套的子状态和转移。

### 2.2 关键字与符号

| 关键字/符号 | 说明 |
|---|---|
| def | 变量定义 |
| int, float | 类型 |
| state | 状态定义 |
| enter, during, exit | 生命周期操作 |
| before, after | 生命周期切面 |
| effect | 转移完毕后执行操作 |
| if | 条件 |
| [*] | 初始/退出状态 |
| -> | 状态转换 |
| ! | 强制转换 |
| : , :: | 事件链、命名空间 |
| = | 赋值 |
| true, false | 布尔常量 |
| pi, E, tau | 数学常量 |
| //, # | 注释 |

## 3. 变量定义

### 3.1 语法格式
```fcstm
def <类型> <变量名> = <初始值>;
```
- 类型：int, float

### 3.2 示例
```fcstm
def int counter = 0;
def float temperature = 25.5;
def int flags = 0xFF;
def float pi_value = pi;
def int result = 5 + 3 * 2;
```

### 3.3 语义规则
- 变量名全局唯一
- 初始值类型需与声明类型兼容
- 不允许包含其他关键字

### 3.4 常见错误
- 重复定义：`def int x = 1; def float x = 2.0;`
- 未声明引用：`def int y = unknown_var;`
- 包含其他关键字：`def int a = 0; def int b = a + 1;`

## 4. 状态定义

### 4.1 语法格式

```fcstm
state <状态名> {
    // 子状态、转移、生命周期操作
}
```

注意，在定义叶子状态时，不能包含子节点。在定义复合状态时，需要包含子节点，并且需要有入口转移

**叶子状态示例：**

```
state Idle;
state Red {
    during {
    a = 0x1 << 2;
    }
}
```

**复合状态示例：**

```fcstm
state Machine {
    state Off;
    state On {
        state Slow;
        state Fast;
    }

    [*] -> Off;
    Off -> On: if [power_switch == 1];
    On -> Off: if [power_switch == 0];
}
```

### 4.3 语义规则

- 命名唯一性：同一作用域下状态名唯一
- 入口转移：复合状态必须有入口转移（[*] -> ...）
- 状态引用有效性：所有转移的目标状态必须已在当前作用域中定义

### 4.4 常见错误
- 缺少入口转移
- 状态名重复
- 转换目标不存在

## 5. 转移定义

DSL 支持三种类型的状态转移，每种具有不同的语法结构：

```fcstm
transition_definition ::= entryTransitionDefinition 
                       | normalTransitionDefinition 
                       | exitTransitionDefinition
```

### 5.1 入口转移（entryTransitionDefinition）

入口转移用于定义进入复合状态时的初始状态。

```fcstm
[*] -> target_state [: chain_id|:: event_name] [if [condition]] [effect { operations }] ;
```
示例：

```fcstm
[*] -> Idle;
[*] -> Running: startup_event;
[*] -> Active: if [initialized == 0x1];
[*] -> Ready effect { counter = 0};
```

### 5.2 普通转移（normalTransitionDefinition）

普通转移用于连接两个命名状态之间的转移关系。

```fcstm
from_state -> to_state [: chain_id|:: event_name] [if [condition]] [effect { operations }] ;
```

示例：

```fcstm
Idle -> Running;
Slow -> Fast : speed_up;
Slow -> Fast :: speed_up; 
Active -> Inactive: if [timeout > 100];
Processing -> Complete effect { result = output; }
```

### 5.3 退出转移（exitTransitionDefinition）

退出转移用于定义离开复合状态的方式。

```fcstm
from_state -> [*] [: chain_id | :: event_name] [if [condition]] [effect { operations }] ;
```

示例：

```fcstm
Error -> [*];
Complete -> [*]: finish_event;
Running -> [*]: if [shutdown_requested];
Active -> [*] effect { cleanup_flag = 0x1; }
```

### 5.4 事件链标识符

事件链标识符（chain\_id）使用点号 `.` 表示层级结构的事件命名空间。

**正确示例：**

```fcstm
State1 -> State2: event_name;
Parent -> Child: parent.child.event;
Active -> Idle: system.shutdown.requested;
```

**说明：**

考虑如下定义：

```fcstm
state Root {
    [*] -> A;
    A -> B : E;
    B -> C : E;
}
```

上述代码表示：A -> B 和 B -> C 都由同一个事件 `Root.E` 触发。如果希望这两个转移由不同的事件触发，则应写作：

```fcstm
state Root {
    [*] -> A;
    A -> B : A.E;
    B -> C : B.E;
}
```

此时，两个事件分别是 `Root.A.E` 和 `Root.B.E`，实现了事件的区分。

为简化这种写法，DSL 还提供了语法糖：

```fcstm
state Root {
    [*] -> A;
    A -> B :: E;  // 这句等价于 A -> B : A.E;
    B -> C :: E;  // 这句等价于 B -> C : B.E;
}
```

这里的 `::` 符号表示使用“源状态名”作为事件前缀，因此此写法等价于上述的 `A.E` 和 `B.E`。

另外，对于事件标识符，还支持另一种绝对路径的形式，以`/` 开头，格式如下

```
state Root {
	[*] -> A;
	A -> B : /Root.A.E;
	B -> C : /Root.B.E;
}
```

这份代码和上述的代码等价，以绝对路径的形式定义了所使用的事件。

对于转移所使用的事件，当两个不同的转移使用了同一个事件的时，将在代码生成阶段使用同一个抽象函数，实现逻辑的时候只用写一遍即可，可以有效实现针对事件触发逻辑的重用。

### 5.5 强制转移

**语法：**

```fcstm
! from_state -> to_state [: chain_id|:: event_name] [if [condition]] [effect { operations }] ;
```

其中，from_state可以是具体的状态，也可以是*（代表ANY），但是不可以是[\*]

转移的to_state可以是具体状态也可以是[\*]

**示例：**

```fcstm
! Running -> Idle : if [emergency == true];
! * -> Error : if [system_failure == true];
```

### 5.6 转移完毕后执行操作（effect）

`effect` 块表示**转移完毕后会执行的操作**

**正确示例：**

```fcstm
Running -> Idle : if [stop == true] effect { count = 0; };
```

上面的示例表示当状态从 `Running` 转移到 `Idle` 后，执行  count = 0; 这段代码。

### 5.7 语义规则

转移定义必须遵守以下语义规则：

-   **状态存在性**：起始状态和目标状态必须在当前作用域中定义
    
-   **变量合法性**：所有条件判断和转移效果操作中使用的变量必须已声明
    
-   **布尔表达式**：条件表达式（guard）必须能被解析为布尔值
    
-   **入口要求**：每个复合状态至少必须包含一个入口转移（`[*] ->`）

### 5.8 常见错误

-   转移引用了未定义的状态
    
-   复合状态缺失入口转移
    
-   条件或操作中引用了未声明的变量
    
-   链式标识符书写格式错误，例如非法的.表达式

## 6. 生命周期操作（enter/during/exit/切面）

生命周期操作是状态在 **进入（enter）**、**停留期间（during）** 和 **退出（exit）** 这三个阶段中执行的代码逻辑。

#### 操作类型

状态支持三个生命周期阶段，每个阶段对应不同的操作定义：

```fcstm
enter_definition  ::= enterOperations | enterAbstractFunc
during_definition ::= duringOperations | duringAbstractFunc
exit_definition   ::= exitOperations  | exitAbstractFunc
```

需要注意其中abstract和non-abstract的区别：

- enterOperations/duringOperations/exitOperations：直接提供针对DSL中定义的变量进行操作，生成代码后无需用户继续补充实现。
- enterAbstractFunc/duringAbstractFunc/exitAbstractFunc：声明一个待实现的操作接口，生成代码的时候需要代码编写者提供实现。

### 6.1 进入操作（Enter Actions）

进入操作在状态被激活（进入）时执行。

**enter actions语法：**

```fcstm
enter [函数名] { <操作语句>; ... }
```

- 函数名 可省略，也可填写，用于生成代码时作为方法名或标识。

- { ... } 内为实际操作内容。

**示例：**

```fcstm
enter onEnterActive {
    start_time = 0;
    status = 1;
}

enter {
    counter = 0;
    status_flag = 0x1;
}
```

**抽象函数语法：**

```fcstm
enter abstract [函数名] [/* 注释 */];
```

**抽象函数示例：**

```fcstm
enter abstract initialize_system;

enter abstract setup_resources /*
    Initialize system resources
*/
```

 注释部分 `/* xxx */` 是 `abstract enter` 声明的一部分，并可在代码生成阶段被保留或利用。

### 6.2 执行中操作（During Actions）

在状态处于激活状态时执行 `during` 操作。对于复合状态，还支持前置（before）和后置（after）切面操作。

**常规 during 操作语法（仅适用于叶子状态）：**

```fcstm
during [函数名] { <操作语句>; ... }
```

**示例：**

```fcstm
during {
    heartbeat_counter = heartbeat_counter + 1;
}
```

**前置切面语法（仅适用于复合状态）：**

```fcstm
during before [函数名] { <操作语句>; ... }
```

**示例：**

```fcstm
during before {
    update_counters = update_counters + 1;
}
```

**后置切面语法（仅适用于复合状态）：**

```fcstm
during after [函数名] { <操作语句>; ... }
```

**示例：**

```fcstm
during after {
    cleanup_temporary = 0x1;
}
```

**抽象函数语法**

```
during abstract [<函数名>] [/* 注释 */];
```

**示例：**

```fcstm
during abstract abstract_during;

during abstract abstract_during_with_comment /*
    This is the comment
*/
```

**注意**：类似地，注释部分 `/* xxx */` 是 `abstract during` 声明的一部分，在代码生成阶段也会被使用。

**\>\> during切面操作**

使用\>\> during切面操作可以为当前状态的一颗子树添加during leaf的前后置行为。

**示例：**

```
>> during before {
       a = 1;
}

>> during before abstract_during_befor_with_comment /*
    This is the comment
*/

>> during after {
    a = a + 1;
}
```

### 6.3 退出操作（Exit Actions）

退出操作在状态被离开时执行。

**exit actions语法：**

```fcstm
exit [函数名] { <操作语句>; ... }
```

**具体操作示例（Concrete Operations）：**

```fcstm
exit {
    save_state = current_value;
    cleanup_flag = 0x1;
}
```

**抽象函数语法：**

```fcstm
exit abstract [<函数名>] [/* 注释 */];
```

**抽象函数示例（Abstract Functions）：**

```fcstm
exit abstract cleanup_resources;

exit abstract finalize_operations /*
    Clean up before exit
*/
```

 **注意**：`/* xxx */` 注释是抽象退出函数定义的一部分，在代码生成阶段也会被使用。

### 6.4 语义规则

生命周期操作必须遵循以下语义约束：

-   **变量合法性**：所有引用的变量必须在当前上下文中已声明
    
-   **切面限制**：`before` 和 `after` 切面仅适用于复合状态
    
-   **赋值目标**：只能对已声明的变量进行赋值操作
    
-   **类型一致性**：赋值表达式的左右两侧类型必须兼容

### 6.5 常见错误

**错误示例：**

-   赋值给未声明的变量（语义错误）
-   在叶子状态中使用 `before` 或 `after` 切面（语义错误）
-   不兼容类型之间的赋值（语义错误）
-   抽象函数声明格式错误（语法错误）

## 7 表达式

DSL 支持丰富的表达式类型，用于数学计算与逻辑运算：

### 7.1 常量

**整数常量：**

-   十进制：`42`，`0`，`123`
    
-   十六进制：`0xFF`，`0x2A`，`0x100`
    

**浮点常量：**

-   标准形式：`3.14`，`2.5`，`0.1`
    
-   科学计数法：`1e6`，`2.5e-3`，`1E10`
    

**数学常数：**

-   `pi`，`E`，`tau`

**布尔常量（Boolean Literals）：**

-   真值：`true`，`True`，`TRUE`
    
-   假值：`false`，`False`，`FALSE`
    

### 7.2 运算符

**算术运算符（按优先级排列）：**

-   括号：`()`
    
-   一元运算：`+`，`-`
    
-   幂运算：`**`
    
-   乘除取余：`*`，`/`，`%`
    
-   加减运算：`+`，`-`
    

**比较运算符：**

-   关系运算：`<`，`>`，`<=`，`>=`
    
-   相等运算：`==`，`!=`
    

**逻辑运算符：**

-   一元逻辑：`!`，`not`
    
-   二元逻辑：`&&`，`||`，`and`，`or`
    

**按位运算符：**

-   按位与、或、异或：`&`，`|`，`^`
    
-   位移运算：`<<`，`>>`
    

### 7.3 数学函数

DSL 提供了丰富的数学函数支持：

**三角函数：**

-   基本：`sin(x)`，`cos(x)`，`tan(x)`
    
-   反函数：`asin(x)`，`acos(x)`，`atan(x)`
    
-   双曲函数：`sinh(x)`，`cosh(x)`，`tanh(x)`
    

**指数与对数：**

-   `exp(x)`，`log(x)`，`log10(x)`，`log2(x)`

**其他数学函数：**

-   `sqrt(x)`，`abs(x)`，`ceil(x)`，`floor(x)`，`round(x)`

### 7.4 条件表达式

条件表达式使用三元运算符语法：

**语法格式：**

```fcstm
(条件) ? 真值 : 假值
```

**正确示例：**

```fcstm
(x > 0) ? 1 : -1
(temperature > 25.0) ? "hot" : "cold"
(count == 0) ? initial_value : current_value
```

注意**：条件部分必须显式使用圆括号 `()` 包裹。

### 7.5 语义规则

表达式必须满足以下语义约束：

-   **变量声明**：所有引用的变量必须先声明
-   **类型一致性**：运算中的操作数必须是兼容类型
-   **函数参数**：数学函数必须传入合法的参数类型
-   **数值表达式和布尔表达式需严格区分**，不允许像常见编程语言一样混用：赋值一定是数值表达式，三目运算符的condition或if [condition]表达式一定是布尔表达式

### 7.6 常见错误

-   使用了未声明的变量
-   数值表达式和布尔表达式混用
-   函数参数非法，如类型错误或缺失参数
-   条件表达式语法错误，如缺少括号或冒号

**错误示例：**

```
a = 10 > 5 ;	(赋值应为数值表达式)
if [x] ;	(if [condition]中的condition需为布尔表达式)
a = b > 0 ? 1 : -1;		(应写为a = (b > 0) ? 1 : -1) )
```

## 8 赋值操作

### 8.1 语法格式

```fcstm
变量 = 表达式;
```

**适用场景：**

用于 `effect` 块和生命周期操作中的标准赋值语句。

### 8.2 正确示例

**在 Effect 块中使用：**

```fcstm
State1 -> State2 effect {
    counter = counter + 1;
    status = 1;
    result = sin(angle) * radius;
}
```

**在生命周期操作中使用：**

```fcstm
enter {
    initialization_flag = 0x1;
    start_time = current_time;
}
```

### 8.3 语义规则

操作语句必须遵循以下语义规则：

-   **变量存在性**：被赋值的目标变量必须已经声明
    
-   **表达式合法性**：右侧表达式必须是合法的、可求值的表达式
    
-   **类型兼容性**：赋值时变量类型与表达式类型必须兼容
    
-   **作用域有效性**：变量必须在当前上下文中可访问

### 8.4 常见错误

## 10 语义验证规则

### 10.1 综合验证

在解析（Parsing）过程中，DSL 解析器会自动执行多方面的语义验证，包括变量、状态、表达式和结构等验证内容：

**变量验证**

-   程序中所有变量名必须全局唯一
    
-   所有引用的变量必须已声明
    
-   赋值和表达式中的类型必须保持一致
    

**状态验证**

-   每个作用域中的状态名必须唯一
    
-   所有转移中的状态引用必须有效
    
-   每个复合状态必须至少定义一个入口转移（`[*] -> state`）
    

**表达式验证**

-   数学表达式与逻辑表达式必须结构完整、语法正确
    
-   所有函数调用必须提供合法且匹配的参数
    
-   操作符的优先级和结合性必须符合语义规范
    

**结构验证**

-   复合状态的嵌套结构必须合理
    
-   生命周期操作（enter、during、exit）必须放置在正确的状态中
    
-   转移的连接结构必须有效（例如源状态和目标状态必须存在）
    

### 10.2 错误处理

解析器会为常见错误提供详细的错误信息，帮助开发者快速定位问题：

**语法错误**

-   表达式或语句格式不正确
    
-   缺失标点符号或关键字（如 `;`、`{}`）
    
-   非法的符号序列（token sequence）
    

**语义错误**

-   引用了未声明的变量或未定义的状态
-   运算中存在类型不匹配（如布尔值与数值混用）
-   状态结构不一致（如重复嵌套、自引用等）

## 11. 最佳实践

本章汇总了在实际建模和代码生成过程中推荐的一些用法，包括状态层级结构设计、事件组织方式、条件逻辑模式以及资源管理策略，供开发者参考使用。

### 11.1 层级状态组织

通过层级分解方式组织复杂状态机，提高可读性与模块化程度：

```plaintext
state SystemController {
    state PowerManagement {
        state LowPower;
        state NormalPower;
        state HighPower;
    }

    state OperationalMode {
        state Idle;
        state Processing {
            state DataInput;
            state Computation;
            state OutputGeneration;
        }
    }
}
```

### 11.2 事件驱动转移

使用**事件链标识符（chain identifier）**建立清晰的事件层级结构：

```fcstm
ProcessingState -> IdleState: user.input.complete;
IdleState -> ProcessingState: system.timer.expired;
ErrorState -> RecoveryState: error.recovery.initiated;
```

### 11.3 条件逻辑模式

使用条件表达式实现复杂的决策逻辑：

```fcstm
Active -> Standby: if [(battery_level < 20) && (charging_state == 0x0)];
Normal -> Emergency: if [temperature > critical_temp || pressure > max_pressure];
```

### 11.4 资源管理

借助生命周期操作（enter / during / exit）实现资源初始化、监控与清理：

```fcstm
state ResourceManager {
    enter {
        resource_count = 0;
        allocation_flag = 0x1;
    }

    during {
        heartbeat_counter = heartbeat_counter + 1;
    }

    exit {
        cleanup_flag = 0x1;
        resource_count = 0;
    }
}
```

### 11.5 变量与抽象的使用

此外，还需要明确一些模型设计上的思想：

* 通过`def`定义的变量属于状态机模型的一部分
  * 这些变量只可以在DSL所定义的操作中进行修改
  * 严格禁止被外部的任何函数进行修改
* 务必确保所有的抽象操作在被执行后都不会改变变量的值

只有满足上述的性质，才能确保状态机的实际运行效果和DSL定义具有一致性。也只有满足这一点，才能在未来进行更深一步的研究与开发，比如形式化验证等。

        */

        }

    Idle -> [*];
}
